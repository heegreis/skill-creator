# skill-creator 稽核與優化路線圖

## 背景

本計畫針對 `.agents/skills/skill-creator/`，依據 `docs/reference/agentskills/`
中五份規範文件 (`quickstart`、`best-practices`、`optimizing-descriptions`、
`evaluating-skills`、`using-scripts`) 進行對齊稽核，並提出具體重構步驟。

稽核基礎文件:
- `quickstart.md`: SKILL.md 最小結構、progressive disclosure 機制。
- `best-practices.md`: 500 行/5000 token 建議、gotchas、模板、校驗迴圈。
- `optimizing-descriptions.md`: 1024 字元硬上限、train/validation split、
  trigger-rate 偵測需可替換為非 Claude agent。
- `evaluating-skills.md`: `evals/evals.json` 使用 `assertions`、
  `grading.json` 使用 `assertion_results`、`benchmark.json delta` 為數值。
- `using-scripts.md`: 腳本應自含依賴 (PEP 723)、非互動、`--help`、
  結構化輸出、相對路徑。

## 一、對齊落差總覽

| # | 項目 | 規範文件 | 現行 skill-creator | 影響 |
|---|------|---------|-------------------|------|
| 1 | 測試項目欄位名 | `assertions` (evaluating-skills L148-164) | `expectations` (references/schemas.md L20-24, grader.md, aggregate_benchmark.py) | 與官方 schema 不一致，`viewer` 與生態工具不可互換 |
| 2 | `grading.json` 頂層鍵 | `assertion_results` (evaluating-skills L176-205) | `expectations` | 同上 |
| 3 | `benchmark.json.run_summary.delta` | 數值 (`"pass_rate": 0.5`) | 字串 (`"+0.50"`) (aggregate_benchmark.py L218-222) | 下游統計處理錯誤 |
| 4 | `benchmark.json.runs[].eval_name` | 必填 (schemas.md 自述 L252, 但 aggregate 未寫入) | 缺少 | viewer 顯示空白 |
| 5 | `SKILL.md` 行數 | <500 行 (best-practices L92-94) | 510 行 | 佔用 context，違背 progressive disclosure |
| 6 | 跨平台腳本執行 | 規範不綁定平台 | `run_eval.py` 使用 POSIX-only `select`/`os.read(fd)`、SKILL.md 內含 `nohup`、`$!`、`kill $PID`、`/tmp` | Windows (orbits 主要環境) 執行失敗 |
| 7 | 觸發偵測可替換性 | optimizing-descriptions L98-115 示範用抽象 `check_triggered` | 硬寫 `claude -p` 子程序、`Skill`/`Read` 工具名、`.claude/commands/` | 無法用於 Kilo、Copilot、Codex |
| 8 | 腳本自含依賴 | PEP 723 inline metadata (using-scripts L137-171) | 依賴隱性 `yaml`、無 shebang、無 `# /// script` block | 安裝流程不明、易失敗 |
| 9 | 腳本執行方式 | `python scripts/xyz.py` (using-scripts L118-131) | 需 `python -m scripts.xyz` (因 `__init__.py` + 相對 import) | 違背規範的相對路徑慣例 |
| 10 | `--help`、錯誤訊息、結構化輸出 | using-scripts L268-319 | 大多腳本無 `--help` 範例、錯誤只有 `Warning: …` 普通字串、mixed stdout/stderr | 代理難以從錯誤自動修正 |
| 11 | `evals.json` 有 `assertions` 欄位 | evaluating-skills L148-164 | schemas.md 顯示 `expectations`；quickstart 與 SKILL.md 寫 `assertions`; 內部自相矛盾 | 用戶無所適從 |
| 12 | `history.json` 欄位 `expectation_pass_rate` | 規範無此檔 (內部機制) | 使用 `expectation_`，應更名為 `assertion_pass_rate` 以一致 | 術語分裂 |
| 13 | SKILL.md body 長度檢查 | best-practices L92-94 | `quick_validate.py` 僅檢 frontmatter | 容易破格而無警告 |
| 14 | 描述 lint | 1024 字元 (optimizing-descriptions L29) | `quick_validate.py` 已檢；OK | — |
| 15 | SKILL.md 自身的 `compatibility` frontmatter | using-scripts L100 | 未標注，但需 Python + `claude` CLI | 使用者難判斷環境 |
| 16 | 繁體中文文件偏好 | orbits `AGENTS.md` | SKILL.md 全英、`Cool? Cool.` 等口語，未翻 | 違反專案文件規範 |
| 17 | 未提供自身 evals | evaluating-skills L44 | `evals/` 不存在 | 無法自測 meta-skill |
| 18 | `feedback.json` schema | evaluating-skills L265-270 簡易 `{eval: feedback}` | skill-creator 使用 `{"reviews":[{run_id,feedback,timestamp}], status}` | 與示範規範不一致 (可保留但需文件說明) |

## 二、優化路線圖 (分階段)

### 階段 0 — 不改碼，先定案

目的: 在動刀前鎖定介面契約，避免二次重構。

- [ ] 確認 `assertions` vs `expectations` 最終採用 **`assertions`** (遵循官方文件)。
- [ ] 確認 `grading.json` 頂層採用 **`assertion_results`**。
- [ ] 確認 `benchmark.json.delta` 改為 **數值**。
- [ ] 確認是否於本輪同步翻譯 SKILL.md 為繁體中文 (需用戶同意)。
- [ ] 確認觸發偵測抽象層是否要支援 Kilo (本專案主 agent)。

### 階段 1 — 結構與術語對齊 (低風險、高影響)

1. **統一欄位名稱 (falls #1, #2, #11, #12)**
   - `references/schemas.md`:
     - `evals.json`: `expectations` → `assertions`
     - `grading.json`: 頂層 `expectations` → `assertion_results`、
       `summary` 保持、但 `user_notes_summary` 不變
     - `history.json`: `expectation_pass_rate` → `assertion_pass_rate`
   - `agents/grader.md`: 全檔 `expectation(s)` → `assertion(s)`、
     輸出 JSON 範例對齊
   - `scripts/aggregate_benchmark.py`:
     - 讀取時同時接受新舊欄位 (過渡期向後相容)
     - 輸出 `runs[].assertion_results` 與 `runs[].eval_name`
   - `eval-viewer/generate_review.py`、`eval-viewer/viewer.html`:
     - 同步改讀新欄位，並保留舊欄位 fallback 一段期間
   - `SKILL.md`: 所有行內 `assertions` / `expectations` 混用處統一。

2. **`benchmark.json.delta` 改為數值 (fall #3)**
   - 修改 `aggregate_benchmark.py::aggregate_results`:
     ```python
     run_summary["delta"] = {
         "pass_rate": round(delta_pass_rate, 4),
         "time_seconds": round(delta_time, 2),
         "tokens": round(delta_tokens),
     }
     ```
   - `generate_markdown`: 輸出時用 `f"{v:+.2f}"` 格式化字串，而非存在 JSON 中。
   - Viewer 若依賴字串格式，同步調整。

3. **補齊 `eval_name` (fall #4)**
   - `aggregate_benchmark.py::load_run_results`: 從 `eval_metadata.json.eval_name`
     取值，fallback 為 `eval-{id}`。
   - 在 `runs[]` 物件中一併寫入 `eval_name`。

### 階段 2 — 跨平台與可替換 (中風險)

4. **Windows 相容 (fall #6)**
   - `run_eval.py::run_single_query`:
     - 以 `subprocess.Popen(..., bufsize=1, text=True)` + 迴圈讀 `stdout.readline()`
       取代 `select.select` / `os.read(fd)`
     - 以 `threading.Timer` 或 `process.wait(timeout=...)` 實作逾時
     - 將子程序終止改為 `process.terminate()` 並於 Windows 使用
       `CREATE_NEW_PROCESS_GROUP` 搭配 `CTRL_BREAK_EVENT`
   - SKILL.md: 將 bash one-liner 章節包裝成「POSIX 指令」與「PowerShell 指令」
     分頁塊 (參考 `using-scripts.md` 的 `<Tabs>` 風格)
     - `nohup … &` → PowerShell `Start-Process -WindowStyle Hidden`
     - `$VIEWER_PID` / `kill $VIEWER_PID` → 儲存於 `.viewer_pid` 檔並以
       `Stop-Process -Id (Get-Content .viewer_pid)`
     - `/tmp/eval_review_*.html` → 使用 `tempfile.gettempdir()`，指令示範用
       `$env:TEMP`

5. **觸發偵測抽象層 (fall #7)**
   - 於 `scripts/` 新增 `trigger_adapters/` 子目錄:
     - `base.py`: 定義 `class TriggerAdapter` 介面 (`check_triggered(query, skill_name, description, timeout) -> bool`)
     - `claude_code.py`: 現行 `run_single_query` 的邏輯遷入
     - `kilo.py`: 骨架 (待 Kilo CLI 支援時填)
     - `manual.py`: 互動式降級 (供 Claude.ai 等環境)
   - `run_eval.py`:
     - 新增 `--adapter {claude-code,kilo,manual}` 參數，預設 `claude-code`
     - `run_eval()` 透過 adapter 工廠取得實作
   - SKILL.md 的 Step 3 / Description Optimization 章節補充如何指定 adapter。

### 階段 3 — 腳本介面現代化 (中風險)

6. **腳本可執行性 (fall #8, #9)**
   - 為每支 CLI 腳本 (`run_loop.py`, `run_eval.py`, `aggregate_benchmark.py`,
     `package_skill.py`, `quick_validate.py`, `improve_description.py`,
     `generate_report.py`) 加入:
     ```python
     #!/usr/bin/env -S uv run --script
     # /// script
     # requires-python = ">=3.10"
     # dependencies = ["pyyaml"]  # 視腳本需求
     # ///
     ```
   - 解除 `scripts.*` 模組式執行限制:
     - 將 `from scripts.utils import parse_skill_md` 改為套件相對 import
       (`from .utils import …`) 或保留 `scripts/__init__.py` 但同時支援
       `python scripts/run_loop.py` (加入 `sys.path` bootstrap)
     - 或改用同目錄 flat import 並移除 `__init__.py` 依賴
   - SKILL.md 所有 `python -m scripts.xxx` 範例改為
     `uv run scripts/xxx.py` 與 `python scripts/xxx.py` 雙版本。

7. **`--help`、錯誤、結構化輸出 (fall #10)**
   - 於 `argparse` 加 `description`、`epilog` 含使用範例 (遵循 using-scripts L272-285)。
   - 失敗訊息格式統一: `Error: <what> — expected <x>, got <y>. Try: <hint>`
   - 將 `print(f"Warning: …")` 分流到 `stderr`，並讓 stdout 僅輸出 JSON/資料。
   - `aggregate_benchmark.py` 目前混用 stdout print 與寫檔; 改為:
     - stdout: 乾淨的 JSON 摘要 (或只在 `--quiet` 關閉)
     - stderr: 進度訊息 / warnings
   - 為破壞性或檔案寫入腳本加入 `--dry-run`。
   - 加入 `--output -` 允許串接管線。

### 階段 4 — 驗證、文件與自測 (低風險)

8. **`quick_validate.py` 擴充 (fall #13, #15)**
   - SKILL.md body lint:
     - 警告 line count > 500 (非致錯)
     - 警告 token 預估 > 5000 (以 `len(content)/4` 粗估)
     - 警告無 `##` 標題結構
   - 檢查 `scripts/` 中被 SKILL.md 引用的檔案皆存在。
   - 檢查 `references/`、`assets/` 引用路徑存在。
   - 加入 `--strict` 將 warning 升級為 error。

9. **SKILL.md 瘦身 (fall #5)**
   - 將「Cowork-Specific Instructions」、「Claude.ai-specific instructions」
     與「Advanced: Blind comparison」搬移至
     `references/environments.md` 與 `references/blind-comparison.md`。
   - 在主文件保留單行指向: e.g.
     「若在 Cowork/Claude.ai，請先閱讀 `references/environments.md`」
   - 目標: SKILL.md ≤ 400 行，符合 progressive disclosure。

10. **自身 SKILL.md 加上 `compatibility` (fall #15)**
    ```yaml
    compatibility: >
      Requires Python 3.10+, optional `uv` for self-contained scripts,
      and the `claude` CLI for Claude Code trigger evaluation. The eval
      viewer needs a browser; in headless envs use `--static`.
    ```

11. **建立 skill-creator 自身的 evals (fall #17)**
    - `.agents/skills/skill-creator/evals/evals.json`: 3 個測試:
      1. 「幫我把這個 bash pipeline 包裝成 skill」(should trigger)
      2. 「Optimize the description of my existing `pdf-tools` skill」(should trigger)
      3. 「Run my failing pytest」(should not trigger)
    - `assertions` 採新 schema，做為本次重構的迴歸測試。

12. **繁體中文化 (fall #16) — 可選**
    - 若用戶同意，於 `references/` 增 `skill-guide-zh-TW.md`，SKILL.md 主體仍保留英文
      (因 skill 本身為 Anthropic 上游；若 fork 為本地版再全文翻譯)。
    - 或僅翻譯本專案維護文件與 commit message，保留上游 SKILL.md 原文。

### 階段 5 — 驗收

- 執行 `quick_validate.py --strict` 通過。
- 執行 `aggregate_benchmark.py` 對既有 workspace 樣本驗證不報 schema warning。
- 執行 `run_eval.py --adapter claude-code --dry-run` 於 Windows 不 crash。
- 手動跑自身 evals，trigger rate ≥ 0.67 for should-trigger、≤ 0.33 for should-not-trigger。
- 以 `package_skill.py` 打包，確認新 schema 向後相容舊 workspaces。

## 三、實作順序建議

1. 階段 0 (決策) → 階段 1 (改 schema/rename，單次大 PR，附資料遷移腳本)
2. 階段 2 (跨平台、adapter) — 需 Windows 實機測試
3. 階段 3 (腳本介面) — 可與階段 4 並行
4. 階段 4 (驗證/文件/自測)
5. 階段 5 (驗收)

每階段獨立 commit，PR 描述需附遷移說明 (尤其階段 1 的欄位改名，
舊 workspace 如何升級)。

## 四、用戶已確認的決策

1. **Schema 改名**: 採硬切換 + 一次性遷移腳本 `scripts/migrate_schema.py`
   (輸入舊 workspace 目錄，原地升級 `grading.json` 與 `benchmark.json`，
   寫入 `.schema-migrated` 標記避免重複)。
2. **Adapter 抽象**: 本輪僅抽介面 + `claude-code` 實作；
   `kilo.py`、`manual.py` 僅留骨架與 `NotImplementedError`。
3. **繁中化**: 保留 SKILL.md 英文原文；新增
   `references/skill-guide-zh-TW.md` 作為繁中導覽 (涵蓋核心流程摘要、
   Windows 指令對照、常見 gotchas)。
4. **執行範圍**: 全部階段 0-5 均納入本輪實作。
5. **自身 evals**: 採 `claude-code` adapter (與主力相同)。
